## Group 2: `action_config` key mismatch — NgActionWrapper vs original action (5 failures)

**Failing tests**:
- `tests/config/test_store_action_radio.py`
- `tests/config/test_store_action_slider.py`
- `tests/config/test_store_action_file.py`
- `tests/config/test_store_const_action_toggle.py`
- `tests/config/test_action_validation.py`

**Root cause**: `_NgActionsContainer.add_argument` creates an `NgActionWrapper` via `_copy_as_type` (shallow copy of the action), but the *original* action object is what argparse stores in `_actions`. When tests write:

```python
parser.nicegooey_config.action_config[ng_action_wrapper] = NiceGooeyConfig.ActionConfig(element_override=...)
```

the dict key is the `NgActionWrapper` instance. When the UI renders, `from_action` receives the *original* action object from `parser._actions`, and `get_action_config(original_action)` finds nothing because the dict was keyed by the wrapper, not the original. The `element_override` / `validation` is silently ignored and defaults are used instead.

Note: tests using the `.nicegooey_config` property *setter* (e.g. `action.nicegooey_config = ...`) work correctly because the setter explicitly uses `self.original_action` as the key. Only direct dict assignment (`action_config[wrapper] = ...`) is broken.

**What to fix**:

Update `NiceGooeyConfig.get_action_config` to do a two-pass lookup — first by the action directly, then by scanning for `NgActionWrapper` keys whose `.original_action` matches the incoming action.

## Group 3: `ui.number(precision=N)` prop not exposed (2 failures)

**Failing tests**:
- `tests/config/test_number_precision_float.py`
- `tests/config/test_number_precision_int.py`

**Root cause**: `ui.number(precision=N)` passes `precision` to NiceGUI's constructor, but NiceGUI does not store it as a named Quasar prop in `element._props`. The tests check `number_el.props.get("precision") == 2` and get `None`.

The current code in `action_ui_element.py`:
```python
basic_element = ui.number(precision=0 if number_precision is None else number_precision).props("dense")
```
The `precision=N` Python kwarg is not forwarded to the Quasar component as a prop.

**What to fix**: Chain `.props(f"precision={N}")` explicitly after the `ui.number(...)` call so that `precision` is stored as a named Quasar prop. For the `int` case, always emit `precision=0`; for `float`, emit `precision={number_precision}` when set.

## Group 4: TypeError from `number_precision` validation not propagating (2 failures + teardown ERRORs)

**Failing tests**:
- `tests/config/test_number_precision_on_choices_raises.py`
- `tests/config/test_number_precision_on_non_numeric_raises.py`

**Root cause**: `_render_action_single` raises `TypeError("... number_precision ...")` when `number_precision` is set on an incompatible action (choices or non-numeric). This error fires *inside* NiceGUI's sub-page route handler, which catches it and returns a 500 response — it never propagates to `await user.open("/")`. The tests wrap `user.open("/")` in `pytest.raises(TypeError)`, which never fires because the exception is swallowed by NiceGUI.

**What to fix**: Move the `number_precision` compatibility check *before* the rendering phase — for example, validate it at the point where `nicegooey_config` is assigned (in the setter or in a dedicated check that runs before `ui.run`). This ensures the `TypeError` propagates during `await user.open("/")` before NiceGUI's routing can catch it.

## Group 5: `ValidationCheckbox` MRO — missing `validation` argument (1 failure + teardown ERROR)

**Failing test**: `tests/config/test_override_type_renders_bool_input.py`

**Root cause**: `ValidationCheckbox(ui.checkbox, ValidationElement)` — the Python MRO means that `super().__init__(text, value=value, on_change=on_change)` in `ValidationCheckbox.__init__` eventually reaches `ValidationElement.__init__()` without the required `validation` positional argument:

```
TypeError: ValidationElement.__init__() missing 1 required positional argument: 'validation'
```

This raises inside the NiceGUI sub-page handler, producing a 500 page instead of the expected `ValidationCheckbox` widget.

**What to fix**: `ValidationCheckbox.__init__` must supply `validation` when cooperating with the MRO. Either:
- Call `ValidationElement.__init__(self, value=value, validation={})` explicitly, or
- Pass `validation={}` (or `validation=None`) through `super().__init__` using cooperative multiple-inheritance kwargs.

## Group 6: `append_const` enable checkbox never rendered (1 failure)

**Failing test**: `tests/config/test_append_const_enable_checkbox_default.py`

**Root cause**: `ListActionUiElement._render_action_required` completely overrides `ActionUiElement._render_action_required` and renders only the list widget — it never consults `_should_render_enable_box`. `AppendConstActionUiElement` defines `_should_render_enable_box` to return `False` for non-required actions (indicating an enable checkbox *should* appear), but since `ListActionUiElement`'s override never calls this method, the checkbox is never rendered.

**What to fix**: Add a `_render_action_required` override in `AppendConstActionUiElement` that:
1. Calls `_should_render_enable_box(action_info)`.
2. If `False` (not required → enable box needed): wraps the list rendering in a `ui.row()` with a `ui.checkbox` (marked `ENABLE_PARAMETER_BOX_MARKER`) and a `DisableableDiv` — matching the pattern in `ActionUiElement._render_action_required`.
3. If `True` (required → no enable box): renders the list directly (the current `ListActionUiElement` behavior).

## Group 7: `choices` + `nargs="?"` + `const` initializes to `default` instead of `const` (1 failure)

**Failing test**: `tests/basic_actions/test_string_action_with_choices_const.py`

**Root cause**: In `_render_input_element`, for a `choices` action with `nargs="?"` and a `const`, the choices-fallback logic is:

```python
el.value = action_info.action.const   # "orange"
if el.value is None or el.value not in el.options:
    el.value = action_info.action.default  # "apple"
```

Before this runs, `nargs_wrapper_element.value = const()` is called on the `OptionalValidationElement`. Its `value.setter` treats `"orange" == none_value` as "no value selected", which sets `checkbox=False` and clears `inner_element.value = ""`. The subsequent `el.value = "orange"` sets the MaxWidthSelect, but if `el.value not in el.options` evaluates to `True` at that point (e.g. because `el.options` stores options in a non-string Quasar format, causing the membership check to fail), the fallback fires and `el.value = "apple"` is set and synced to the namespace.

**What to fix**: Harden the choices initialization so `const` is reliably used as the initial value for `nargs="?"` actions. The `el.value not in el.options` guard should not trigger for a valid const. Possible approaches:
- Fix the `el.options` membership check to compare correctly (e.g. extract labels/values if options are stored as dicts).
- Or restructure so the `OptionalValidationElement` is not prematurely cleared before the choices block, e.g. by setting the inner element's value directly first and then syncing the wrapper.

## Group 8: `test_submit_exception` — business logic runs during page setup (1 ERROR at setup)

**Failing test**: `tests/test_submit_exception.py`

**Root cause**: `nice_gooey_argparse_main` works correctly in production because `ui.run()` blocks forever — code written after `parser.parse_args()` in `main()` never runs until Submit is clicked. In pytest mode, `ui.run()` returns immediately (it just calls the page-setup function), so `parse_args()` also returns immediately, and execution continues:

```python
parser.parse_args()                          # returns immediately in test mode
raise RuntimeError("test error — ...")       # fires during setup, not on Submit
```

This `RuntimeError` propagates out of the test framework's page-setup call, crashing the test at `setup` before the user ever sees the page.

**What to fix**: After `ui.run()` returns in pytest mode, `NiceGooeyMain.parse_args` should stop further execution in the caller via a sentinel mechanism — e.g. raise a private `_NiceGooeySetupComplete` exception. The `nice_gooey_argparse_main` wrapper should catch this exception specifically (distinguishing it from the Submit-time call where `is_running=True` and `parse_args` returns the namespace without raising), and swallow it so setup completes cleanly. On Submit, `main_func()` is called again with `is_running=True`, `parse_args` returns the namespace, the business logic runs, the `RuntimeError` is raised and caught by `submit_xterm_dialog`.
